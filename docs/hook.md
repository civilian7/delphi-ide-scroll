# 후킹(Hooking)과 IDEScroll 개발 이야기

> 한국어 | [English](hook.en.md)

이 문서는 두 가지를 다룹니다.

1. **후킹이란 무엇인가** — Windows 메시지 후킹의 개념과 종류
2. **IDEScroll은 어떻게 만들어졌는가** — 요구사항부터 설계 결정, 구현, 시행착오까지의 과정

## 1. 후킹(Hooking)이란?

### 1.1 개념

**후킹**은 운영체제나 다른 프로그램이 주고받는 **이벤트·메시지·함수 호출을 중간에서 가로채는 기법**입니다. "갈고리(hook)를 걸어 흐름을 낚아챈다"는 비유에서 온 이름입니다.

가로챈 지점에서 우리는 보통 세 가지를 할 수 있습니다.

- **관찰(observe)**: 어떤 이벤트가 일어났는지 들여다본다 (예: 키 로깅, 디버깅)
- **변형(modify)**: 이벤트의 내용이나 결과를 바꾼다 (예: 휠을 다른 방향 스크롤로 변환)
- **차단(suppress)**: 이벤트가 원래 목적지에 도달하지 못하게 막는다

IDEScroll은 이 세 가지를 모두 사용합니다. 휠 이벤트를 **관찰**하고, 디자이너 위라면 스크롤로 **변형**한 뒤, 원본 휠 메시지를 **차단**합니다.

### 1.2 Windows 메시지 후킹

Windows는 GUI가 **메시지(message)** 기반으로 동작합니다. 키 입력, 마우스 이동, 휠 회전 등은 모두 메시지로 만들어져 해당 창의 큐로 전달됩니다.

Windows는 이 메시지 흐름에 끼어들 수 있는 공식 API **`SetWindowsHookEx`** 를 제공합니다.

```pascal
FHookHandle := SetWindowsHookEx(WH_MOUSE_LL, @LowLevelMouseProc, HInstance, 0);
```

- 첫 번째 인자 `WH_MOUSE_LL` 가 **후킹 종류**입니다.
- 두 번째 인자는 이벤트가 발생할 때마다 호출될 **콜백 함수**입니다.
- 후킹을 걸면 같은 종류의 후킹들이 **체인(chain)** 으로 엮이고, 각 콜백은 처리 후 `CallNextHookEx` 로 다음 후킹에게 넘기거나, 0이 아닌 값을 반환해 **이벤트를 소비(차단)** 합니다.

### 1.3 후킹의 종류 — 그리고 핵심 갈림길

후킹 방식은 크게 두 갈래입니다.

| 구분 | 전역 후킹(인젝션 방식) | 저수준 후킹(`WH_*_LL`) |
|---|---|---|
| 예시 | `WH_MOUSE`, `WH_CBT`, `WH_GETMESSAGE` | `WH_MOUSE_LL`, `WH_KEYBOARD_LL` |
| 동작 위치 | 대상 프로세스에 **DLL이 주입**되어 그 프로세스 안에서 실행 | 후킹을 **설치한 프로세스 안**에서 실행 (주입 없음) |
| 제약 | 대상과 **비트수(32/64)가 같아야** 하고 DLL 필요 | 비트수 무관, DLL 불필요 |
| 단점 | 배포·호환성 복잡 | 콜백이 빨리 끝나야 함(시스템 타임아웃) |

이 표의 차이가 IDEScroll 설계를 통째로 결정했습니다. 아래에서 이어집니다.

## 2. IDEScroll 개발 과정

### 2.1 출발점: 요구사항

> "델파이 IDE 폼 디자이너에서, 큰 폼을 디자인할 때 스크롤바가 생긴다. 휠과 Ctrl+휠로 가로/세로 스크롤을 제어하고 싶다. 후킹이 가능한가?"

처음 목표는 **휠 = 가로, Ctrl+휠 = 세로** 였고, 개발 과정에서 **휠 = 세로, Ctrl+휠 = 가로** 로 바뀌었습니다. (그냥 휠을 세로로 두는 것이 더 자연스럽다는 판단)

### 2.2 첫 번째 갈림길: 어떻게 IDE에 끼어들 것인가

IDE 디자이너의 휠 동작을 바꾸려면, 코드가 IDE의 메시지 흐름에 접근해야 합니다. 두 가지 후보가 있었습니다.

**(A) IDE 확장(OpenTools API, `.bpl` 패키지)**
- IDE 안에서 직접 도는 정식 확장 방식
- 그러나 **`.bpl` 은 IDE 버전마다 다시 컴파일**해야 합니다. Delphi는 버전이 매우 많아서, 지원 버전 수만큼 빌드·유지보수가 늘어납니다.
- OTA에는 디자이너 스크롤을 가로채는 인터페이스도 없습니다.

**(B) 외부 프로그램 + 저수준 마우스 훅(`WH_MOUSE_LL`)**
- 별도의 작은 실행 파일이 `WH_MOUSE_LL` 후킹을 설치
- **인젝션이 없으므로** IDE의 버전·비트수와 무관하게 단일 실행 파일로 모든 버전을 커버
- OTA에 의존하지 않음

사용자가 "**여러 버전**을 지원해야 한다"고 못박은 순간, 답은 명확히 **(B)** 였습니다. 1.2~1.3 절의 "인젝션 vs 저수준" 차이가 그대로 결정 근거가 된 것입니다.

### 2.3 두 번째 과제: "어디"를 스크롤할 것인가

휠을 가로챘다 해도, **스크롤 메시지를 어느 창에 보낼지**를 알아야 합니다. 디자이너 표면은 어떤 창일까?

- 스파이 도구(Spy++ 등)로 폼 디자이너 위를 찍어보니 창 클래스가 **`TFormContainerForm`** 으로 나왔습니다.
- 이 클래스는 여러 Delphi/C++Builder 버전에서 **오랫동안 공통으로 쓰여 온 디자이너 컨테이너**라, 다버전 대응에도 유리했습니다.
- 또한 이 창은 `TCustomForm`/`TForm` 의 후손이라, VCL의 `TScrollingWinControl` 이 제공하는 표준 스크롤 처리(`WM_HSCROLL`/`WM_VSCROLL`)에 반응할 가능성이 높았습니다.

그래서 전략이 확정됐습니다.

```
휠 발생
 → 커서 아래 창을 찾고(WindowFromPoint)
 → 부모를 거슬러 올라가며 클래스명이 TFormContainerForm 인 창을 탐색
 → 찾으면 WM_VSCROLL / WM_HSCROLL 을 PostMessage
 → 원본 휠 메시지는 소비(차단)
```

핵심 변환 로직(요지):

```pascal
// 그냥 휠 → 세로, Ctrl+휠 → 가로
LVertical := not ACtrl;

// 휠 위(delta>0) → 위/왼쪽, 아래 → 아래/오른쪽
LBackward := ADelta > 0;
```

### 2.4 세 번째: 메시지 루프와 콜백 안전성

`WH_MOUSE_LL` 콜백은 **후킹을 설치한 스레드의 메시지 펌프** 컨텍스트에서 호출됩니다. 그래서:

- 콜백 안에서는 **무거운 작업을 하지 않습니다** (시스템이 타임아웃을 두기 때문). 스크롤은 동기 `SendMessage` 대신 비동기 `PostMessage` 로 던집니다.
- Ctrl 키 상태는 `GetAsyncKeyState(VK_CONTROL)` 로 물리적 상태를 즉시 읽습니다.
- VCL 앱의 메시지 루프가 콜백을 안전하게 메인 스레드에서 실행해 주므로, UI 갱신(로그 메모)도 별도 동기화 없이 안전합니다.

### 2.5 다듬기: 반복된 사용자 피드백

만드는 과정에서 요구사항이 여러 번 조정됐고, 그때마다 반영했습니다.

- **방향 반전**: 휠=세로, Ctrl+휠=가로 로 변경
- **감도 조정 기능 추가**: 세로/가로 각각 "휠 1노치당 줄 수"를 설정. 값은 `IDEScroll.ini` 에 저장/복원
- **UI**: 체크박스로는 `Enter` 가 안 먹혀, **Default 버튼**으로 바꿔 Windows 표준대로 `Enter` 로 토글되게 함
- **낮은 버전 호환**: 인라인 변수(`var x := ...`, `for var i`)는 신버전 문법이라, 구버전 델파이에서도 컴파일되도록 **전통적인 `var` 블록**으로 전환
- **코드 스타일 정리**: `{$REGION}` 지시어와 `///` XML 문서 주석 제거, 소스는 `src\` 폴더로, 출력은 `bin\`/`dcu\` 로 정리

### 2.6 빌드 중 만난 자잘한 함정

실제로 컴파일을 돌리며 겪고 해결한 것들:

- **`PMSLLHOOKSTRUCT` 미선언**: `Winapi.Windows` 에 이 포인터 타입이 없어, 콜백이 받는 구조체를 직접 선언했습니다.
- **UTF-8 BOM 누락 → 한글 리터럴이 ANSI로 오인(W1057 경고)**: `.pas` 파일에 UTF-8 BOM을 보장해 해결.
- **`.res` 없음**: 빈 리소스 스크립트를 `brcc32` 로 컴파일해 생성.
- **출력 파일 잠김(F2039)**: 테스트로 실행 중이던 `IDEScroll.exe` 가 파일을 잠그고 있어, 프로세스를 종료한 뒤 재빌드.

### 2.7 남은 검증 포인트

`TFormContainerForm` 이 표준 `WM_HSCROLL`/`WM_VSCROLL` 에 실제로 반응하는지는 실행 환경에서 최종 확인하는 항목입니다. `TForm` 후손이라 반응 가능성이 높지만, 만약 자체적으로 그린 스크롤바를 쓴다면 **스크롤바 위치를 직접 조작(`GetScrollInfo`/`SetScrollInfo`)** 하는 폴백으로 전환할 수 있도록 설계 여지를 남겨 두었습니다.

## 3. 정리

IDEScroll은 "작은 편의 기능"이지만, 그 뒤에는 분명한 엔지니어링 판단이 있었습니다.

- **인젝션 후킹이 아니라 저수준 후킹**을 택해 다버전·비트수 문제를 통째로 우회했고,
- **창 클래스명**이라는 안정적인 식별자에 기대 디자이너를 찾았으며,
- **콜백을 가볍게, 메시지는 비동기로** 유지해 시스템 후킹의 안전 규칙을 지켰습니다.

후킹은 강력하지만 그만큼 깨지기 쉬운 기법입니다. "꼭 필요한 최소한만 가로채고, 나머지는 건드리지 않는다"는 원칙이 이 작은 도구를 단순하고 견고하게 만들었습니다.
